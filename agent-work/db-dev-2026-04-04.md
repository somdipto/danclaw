## Findings

### 1. Type-Scheme Alignment

**Status: Minor Drift**

The Phase 1 schema is well-aligned with TypeScript types. Core tables (`users`, `deployments`, `messages`) have exact matches. One drift identified:

- `Activity` interface missing `user_id` field — schema has `user_id UUID NOT NULL` but the TypeScript type does not include it.

### 2. Zod Validator Coverage

**Status: Complete**

All types have corresponding Zod schemas. `activitySchema` needs the same `user_id` fix as the type.

### 3. Migration Inconsistencies

Two versions of the schema exist:
1. `docs/SCHEMA.sql` — canonical, production-ready, includes RLS policies
2. `apps/web/src/sql/migrations/001_initial_schema.sql` — migration file, slightly different naming (`activity_log` vs `activity`)

The migration file is **missing** the `metadata JSONB` column that the API code writes to.

### 4. Mock Data Mismatch

Mock `Activity` objects in `packages/shared/src/mock/index.ts` use:
```typescript
{ id: 'act_01', action: '...', timestamp: '5 min ago', icon: '📧' }
```

But the real schema uses `user_id` and ISO timestamp. Mock data uses relative time strings which won't match the `datetime()` Zod validator.

### 5. Constants vs Schema Duplication

`packages/shared/src/constants/index.ts` defines `DeploymentStatus` type and `DEPLOYMENT_STATUS_COLORS` map. These duplicate the enum values defined in `SCHEMA.sql` CHECK constraints and `deploymentStatusSchema` in validators. Single source of truth issue — constants should import from validators or be the canonical source.

### 6. Missing Indexes

High-priority composite index for message pagination is missing: `(deployment_id, created_at DESC)`.
